# Pudding Widget Development

[中文](widget-development.zh-CN.md)

This guide explains how to build, package, and publish widgets for Pudding Hub.

## What A Widget Is

A Pudding widget is an interactive canvas widget. It has two parts:

- Source files: editable HTML, CSS, and JavaScript under `widgets/<name>/source/`.
- Runtime snapshot: `widgets/<name>/releases/<version>/`, generated by the package script and installed by Pudding.

Pudding installs the runtime snapshot. Maintainers edit the source files and regenerate the snapshot.

## Directory Layout

Create one directory per widget:

```text
widgets/
  <name>/
    manifest.json
    README.md
    assets/
    source/
      index.html
      style.css
      src/
        app.js
        i18n.js
    screenshots/
      .gitkeep
    releases/
      1.0.0/
        manifest.json
        <name>.pudding-widget.json
        assets/
        source/
        screenshots/
```

Use lowercase kebab-case for `<name>`, such as `rps-decider` or `weather-widget`.

`source/index.html` is the entry file. Local CSS and JS referenced from this file are inlined during packaging.

## Manifest

Each widget needs `widgets/<name>/manifest.json`. It keeps widget identity, Hub listing metadata, compatibility, and the current package pointer.

```json
{
  "kind": "pudding.widget.manifest",
  "schema_version": 1,
  "id": "teatak/pudding-hub/widgets/rps-decider",
  "name": "rps-decider",
  "title": {
    "zh-CN": "分歧终端机",
    "zh-TW": "分歧終端機",
    "en": "RPS Decider"
  },
  "version": "1.0.0",
  "description": {
    "zh-CN": "用石头剪刀布快速裁决分歧。",
    "en": "Resolve small disagreements with rock-paper-scissors."
  },
  "icon": "./assets/icon.svg",
  "screenshots": [],
  "tags": ["game", "decision", "multi-session"],
  "size": "lg",
  "orientation": "portrait",
  "author": { "name": "Pudding" },
  "package": "./releases/1.0.0/rps-decider.pudding-widget.json",
  "package_sha256": "...",
  "requires": {
    "widget_api": "^1.0.0"
  }
}
```

`size` accepts `sm`, `md`, or `lg`; omit it to use `lg`. Pudding currently maps `lg` to at least 800x600 for landscape/auto widgets and 600x800 for portrait widgets.

Manifest `icon` is for Hub display only and may point to a local asset such as `./assets/icon.svg`. Package `widget.icon` is also named `icon`, but it is generated by inlining the conventional `assets/icon.svg` file as `data:image/svg+xml;base64,...`, so the generated `.pudding-widget.json` stays self-contained. It must not use `icon_url`, a relative path, or a remote URL.

The package script reads source from the conventional `source/index.html` path. Widget default state belongs in widget code; generated packages use an empty `initial_state` object.

Release snapshots also include compact `releases/<version>/manifest.json` for version auditing.

## Build And Package

Install dependencies:

```bash
pnpm install
```

Package one widget:

```bash
pnpm package-widget rps-decider
```

Build a temporary package for local testing:

```bash
pnpm package-widget rps-decider --dev
```

Package all widgets listed in `widgets/registry.json`:

```bash
pnpm package-widgets
```

The package script:

1. Reads `manifest.json`.
2. Reads `source/index.html`.
3. Inlines local CSS and JS files referenced by the HTML.
4. Minifies the runtime HTML snapshot.
5. Writes `releases/<version>/<name>.pudding-widget.json`.
6. Copies release assets, source, screenshots, and manifest into the release directory.
7. Updates `package_sha256` in `manifest.json` and `widgets/registry.json`.

`--dev` writes only `dev/<name>.dev.pudding-widget.json` and `dev/manifest.json`. It uses `<widget-id>-dev`, so local test installs stay separate from release installs.

## Release Flow

1. Edit files under `source/`.
2. Run `pnpm package-widget <name> --dev`.
3. Import `widgets/<name>/dev/<name>.dev.pudding-widget.json` into Pudding and test it.
4. If testing passes, bump `version` in `manifest.json`.
5. Run `pnpm package-widget <name>` to create the immutable release snapshot.
6. Commit the source, release snapshot, manifest, and registry.

Do not edit files under `releases/<version>/` by hand. Edit `source/`, bump `version` when needed, then package again.

## Runtime API

Widgets communicate with Pudding through `window.pudding`.

### State

```js
const state = window.pudding.getState();

window.pudding.onState((state) => {
  if (!state) return;
  render(state);
});

window.pudding.setState(nextState);
window.pudding.updateState((state) => ({ ...state, count: state.count + 1 }));
```

Use state for the current widget instance, such as board position, form input, current topic, or local UI phase.

State belongs to the widget instance on the canvas. It is not the installed widget package.

### Actions

User clicks should dispatch small actions:

```js
button.addEventListener("click", () => {
  window.pudding.dispatch({ type: "choose_gesture", gesture: "rock" });
});
```

Register one action handler:

```js
window.pudding.onAction((action, context) => {
  const state = window.pudding.getState();

  if (action.type === "choose_gesture") {
    const nextState = applyChoice(state, action, context);
    return { ok: true, state: nextState };
  }

  return { ok: false, error: "unknown_action" };
});
```

Keep actions small. Do not pass the full state through actions.

`context.actor` identifies who triggered the action. `context.visible_sessions` lists currently visible sessions. Pudding provides identity; the widget decides players, seats, turns, and roles.

### Sending Messages To Sessions

Use `window.pudding.send(...)` when a widget needs an AI or user session to receive a message.

```js
window.pudding.send({
  type: "request_gesture",
  to: { sessions: ["s_abc"] },
  text: "Choose rock, paper, or scissors. Call the widget action only.",
  data: { topic: "who washes dishes" },
  lock_until_done: false
});
```

You can also return `send` from an action handler:

```js
return {
  ok: true,
  state: nextState,
  send: {
    type: "announce_result",
    to: { sessions: [winnerSessionID] },
    data: resultPayload,
    lock_until_done: false
  }
};
```

`lock_until_done` defaults to `false`. Set it to `true` only when the widget must block repeat clicks until the target session finishes its response.

### Persistent Widget Data

Use widget data for records that should survive widget instance state resets, such as leaderboards.

```js
const leaderboard = await window.pudding.getData("leaderboard");
await window.pudding.setData("leaderboard", nextLeaderboard);
await window.pudding.deleteData("leaderboard");
```

Use widget instance state for the current game or UI. Use widget data for long-lived widget-owned data.

### Theme

Use Pudding theme tokens instead of hard-coded app colors.

```js
function applyTheme(theme) {
  document.documentElement.dataset.theme = theme?.mode || "dark";
}

applyTheme(window.pudding.theme);
window.pudding.onTheme(applyTheme);
```

Recommended CSS tokens:

```css
body {
  margin: 0;
  color: var(--pudding-fg);
  background: var(--pudding-bg);
  font-family: var(--pudding-font);
}

.panel {
  background: var(--pudding-card);
  border: 1px solid var(--pudding-border);
  border-radius: var(--pudding-radius);
}
```

### Locale

Use `window.pudding.locale` and `window.pudding.onLocale(...)`.

```js
const messages = {
  "zh-CN": { start: "开始" },
  "zh-TW": { start: "開始" },
  en: { start: "Start" }
};

let locale = "zh-CN";

function applyLocale(next) {
  const raw = typeof next === "object" ? next.locale : next;
  locale = raw === "zh-CN" || raw === "zh-TW" || raw === "en" ? raw : "zh-CN";
  document.documentElement.lang = locale;
  render(window.pudding.getState());
}

function t(key) {
  return messages[locale]?.[key] || messages.en[key] || key;
}

applyLocale(window.pudding.locale);
window.pudding.onLocale(applyLocale);
```

Every published widget should include at least English in `title.en` and runtime UI strings.

## LLM-Friendly Design

Pudding widgets are often operated by LLMs. Make the contract boring and explicit:

- Use clear action names: `choose_gesture`, `place_stone`, `reset_game`.
- Use plain payload fields: `gesture`, `row`, `col`, `topic`.
- Avoid ambiguous names like `peer1`, `peer2`, or `result` with different meanings.
- Prefer snake_case for JSON fields in widget state and actions.
- Do not expose hidden scoring or historical patterns to LLM prompts if it can bias future moves.
- Put examples in instructions with exact action shapes.

Good example:

```json
{
  "id": "widget-instance-id",
  "action": {
    "type": "choose_gesture",
    "gesture": "rock"
  }
}
```

## Accessibility And Layout

- Use real buttons for clickable controls.
- Keep text inside fixed-size controls from overflowing.
- Support both light and dark themes.
- Avoid relying on hover-only controls.
- Use `size: "lg"` for most interactive widgets. Use `md` only for compact tools and `sm` only for lightweight read-only surfaces.
- Use `orientation: "portrait"` for tall game boards and `orientation: "landscape"` for wide tools.
- Keep the first screen usable; do not build a landing page for a tool widget.

## Release Checklist

Before publishing:

- `manifest.json` has a stable `id`.
- `title` includes `en` and the target Chinese locales.
- `version` was bumped for user-visible changes.
- `source/` contains the editable implementation.
- `pnpm package-widget <name>` succeeds.
- `releases/<version>/` changed after packaging.
- `manifest.json` and `widgets/registry.json` have updated `package_sha256`.
- The widget works in light and dark themes.
- The widget works with `zh-CN`, `zh-TW`, and `en`.
- The widget does not require external network assets.
